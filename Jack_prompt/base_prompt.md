## 1) โปรเจกต์นี้คืออะไร (ภาพรวม)

**PharmaCare AI** คือแชทบอทสำหรับเภสัชกร ใช้ถาม-ตอบเรื่อง:
- โรคติดเชื้อทางเดินหายใจส่วนบน (เรียกย่อว่า **URI**)
- การใช้ยาปฏิชีวนะ
- ขนาดยา (Dose)

โดย AI จะไปอ้างอิงจากเอกสารจริง 3 แหล่ง (ไม่ได้ตอบมั่ว):

| แหล่งข้อมูล | เนื้อหาคืออะไร |
|--------|---------|
| **AAFP 2022** | แนวทางการใช้ยาปฏิชีวนะรักษา URI (ต่างประเทศ) |
| **URI เด็ก 2562** | แนวทางเวชปฏิบัติของไทย สำหรับเด็ก |
| **Dose supportive** | ตารางขนาดยาและข้อห้ามใช้ (ดึงจากไฟล์ CSV แล้วเปิด PDF หน้าที่ตรงกันได้) |

**เทคโนโลยีที่ใช้:** FastAPI (ฝั่งเซิร์ฟเวอร์) + Qdrant (ฐานข้อมูลค้นหา) + Gemini (AI ของ Google ทำหน้าที่แปลงข้อความเป็นเวกเตอร์ และตอบคำถาม) + หน้าเว็บ HTML/JS ธรรมดา

**วิธีเปิดใช้งาน:** รันผ่าน Docker แล้วเข้าเว็บที่ **http://localhost:8899**

## 3) วิธีรันแอป

### วิธีที่แนะนำ: ใช้ Docker

```bash
docker compose up --build
```

จากนั้นเปิดเว็บที่ **http://localhost:8899**

| จุดไหน | พอร์ต | ความหมาย |
|--------|-------|----------|
| เบราว์เซอร์ของคุณ | **8899** | ใช้พอร์ตนี้เข้าแอป |
| ข้างในคอนเทนเนอร์ (uvicorn) | **8000** | พอร์ตที่เซิร์ฟเวอร์ฟังอยู่ภายใน |
| ใน `docker-compose.yml` | `8899:8000` | คือการแมปจาก host (เครื่องเรา) ไป container |

ถ้าเห็น log ว่า `Uvicorn running on http://0.0.0.0:8000` **ไม่ต้องตกใจ ปกติ**
แค่อย่าไปเปิด `:8000` เอง — ให้เปิด **`:8899`** เท่านั้น

**เรื่องการแก้ไขไฟล์:** โฟลเดอร์ต่อไปนี้ผูก (mount) กับ container ไว้แล้ว แก้แล้วเห็นผลทันที ไม่ต้อง build ใหม่:

| ไฟล์ในเครื่องเรา | ไปอยู่ใน container ที่ |
|------|-----------|
| `./backend` | `/app/backend` |
| `./frontend` | `/app/frontend` |
| `./rag/data` | `/app/rag/data` |
| `./rag/qdrant_db` | `/app/rag/qdrant_db` |
| `./.env` | `/app/.env` |

- ต้อง `docker compose up --build` ใหม่ **เฉพาะเมื่อ**: แก้ `Dockerfile` หรือ `requirements.txt`
- แก้ `backend/` หรือ `frontend/` → แค่ **restart หรือรีเฟรชหน้าเว็บ** ก็พอ

### วิธีรันแบบไม่ใช้ Docker (รันตรงในเครื่อง)

```bash
uvicorn backend.main:app --reload --host 0.0.0.0 --port 8000
# แล้วเปิด http://localhost:8000
```

ต้องมีไฟล์ `.env` ที่มี `GOOGLE_API_KEY` อยู่ด้วย

## 4) เบื้องหลังการทำงานของระบบ

### 4.1) ขั้นตอน "ตัดข้อมูลเป็นชิ้นๆ" (Chunk) จากเอกสาร ไปจนถึงเก็บใน Qdrant

> 💡 **Chunk คืออะไร:** การตัดเอกสารยาวๆ ให้เป็นชิ้นเล็กๆ (เหมือนตัดหนังสือเป็นย่อหน้า) เพื่อให้ AI ค้นหาและดึงมาใช้ตอบคำถามได้ง่ายขึ้น

#### ตารางไฟล์ที่เกี่ยวข้องกับการตัด chunk

| ไฟล์ | ทำหน้าที่อะไร | ใช้ตอนไหน |
|------|--------|-----------|
| `backend/pdf_to_md.py` | แปลง PDF → ข้อความ Markdown (ใช้ OCR ยี่ห้อ Typhoon) พร้อมใส่เครื่องหมาย `<!-- PAGE N -->` บอกเลขหน้า | เมื่อมี PDF ใหม่ หรืออยากทำ OCR ใหม่ |
| `backend/md_chunker.py` | ตัดไฟล์ `.md` (AAFP / URI) เป็นชิ้นๆ ตามวิธี "Strategy C" | ตอนรัน pipeline |
| `backend/dose_chunker.py` | อ่านไฟล์ **Dose CSV** (ตารางยา) → ตัดเป็นชิ้นข้อมูลยา (ไม่ผ่าน OCR หรือ Strategy C) | ตอนรัน pipeline |
| `backend/patient_group.py` | ติดป้ายให้แต่ละชิ้นข้อมูลว่าเป็นของ "เด็ก/ผู้ใหญ่/ทั่วไป" | ถูกเรียกใช้จาก md_chunker |
| `backend/embed_to_qdrant.py` | อ่านไฟล์ `chunks.jsonl` → แปลงเป็นเวกเตอร์ → เก็บลง Qdrant | ตอนรัน pipeline |
| `rag/pipeline.py` | คำสั่งรวม: ตัด chunk จากทุกแหล่ง → รวมเป็นไฟล์เดียว → แปลงเป็นเวกเตอร์ | คำสั่งหลักที่เรารัน |
| `rag/data/AAFP.md`, `URI.md` | เอกสารหลังผ่าน OCR แล้ว | ใช้เป็นอินพุตของ md_chunker |
| `rag/data/Dose supportive.csv` | ตารางยา | ใช้เป็นอินพุตของ dose_chunker |
| `rag/data/chunks.jsonl` | **ผลลัพธ์รวม** — 1 บรรทัด = ข้อมูล 1 ชิ้น (เปิดดูด้วยตาได้) | ใช้ตรวจสอบก่อนแปลงเป็นเวกเตอร์ / ดีบัก |
| `rag/qdrant_db/` | ฐานข้อมูลเวกเตอร์ที่แอปใช้ค้นหาจริงตอนมีคนถาม | ใช้งานจริง (production) |
| `backend/config.py` | บอก path ต่างๆ + รายชื่อไฟล์ MD / Dose CSV | ใช้ทุกขั้นตอน |

**สรุปเรื่อง Dose (ตารางยา):** ไม่ต้องผ่าน OCR ไม่ต้องผ่าน Strategy C — อ่านแถวจากไฟล์ CSV มาแปลงตรงๆ แล้วเขียนรวมกับข้อมูล AAFP/URI ลงไฟล์ `chunks.jsonl`
ดูตัวอย่างได้ที่ `rag/data/chunks.jsonl` (ค้นคำว่า `dose_` หรือ `"source": "Dose"`)

**จำนวนชิ้นข้อมูลตอนนี้ทั้งหมด 229 ชิ้น** แบ่งเป็น: AAFP 38 ชิ้น + URI 94 ชิ้น + Dose 97 ชิ้น

#### ภาพรวมขั้นตอนทั้งหมด (แผนภาพ)

```mermaid
flowchart TD
```

#### ตัวอย่างการตัด chunk แบบง่ายๆ

**A) เอกสาร AAFP / URI (มาจากไฟล์ .md)**

```text
ในไฟล์ .md มีหัวข้อ + ย่อหน้ายาว + ตาราง
        │
        ├─ ตัดลายน้ำ / รูปภาพออก
        ├─ ข้อความที่อยู่ใต้หัวข้อเดียวกัน → รวมเป็นชิ้นเดียว
        ├─ ถ้ายาวเกินไป → หั่นเป็นหลายชิ้น (ให้เนื้อหาทับกันเล็กน้อย)
        ├─ ถ้าเจอตาราง → แยกเก็บเป็นชิ้นตารางต่างหาก
        └─ ติดป้ายกำกับ เช่น กลุ่มผู้ป่วย=เด็ก, เลขหน้า=20
```

ตัวอย่างในไฟล์ `chunks.jsonl` (แบบย่อ):

```text
[แหล่งข้อมูล: AAFP | หน้า: 2 | หัวข้อ: ... > ตาราง]
[บริบท: ...]
<table>...</table>
```

**B) ข้อมูลยา (Dose) — มาจาก CSV คนละวิธีกับข้างบน**

```text
1 แถว = ยา 1 ชนิด เช่น Paracetamol
  ├─ ถ้ามีข้อมูลขนาดยาผู้ใหญ่ → สร้าง 1 ชิ้น (กลุ่ม=ผู้ใหญ่, หน้า=13)
  └─ ถ้ามีข้อมูลขนาดยาเด็ก    → สร้าง 1 ชิ้น (กลุ่ม=เด็ก, หน้า=13)
→ รวมแล้วได้ประมาณ 97 ชิ้น จาก 54 แถว (ไม่ใช่ 54 ชิ้นตรงๆ เพราะบางแถวแตกเป็น 2 ชิ้น)
```

ตัวอย่างในไฟล์ `chunks.jsonl` (แบบย่อ):

```text
[แหล่งข้อมูล: Dose | หน้า: 13 | ชื่อยา: Paracetamol | กลุ่ม: ผู้ใหญ่]
ชื่อยา: Paracetamol
ขนาดยา (ผู้ใหญ่): ...
อ้างอิง: Dose หน้า 13 (Dose supportive.pdf)
```

#### ถ้าต้องแปลงข้อมูลเป็นเวกเตอร์ใหม่ (re-embed) หรือทำ index ใหม่

ให้ปิดแอปก่อน (กันฐานข้อมูลล็อก) แล้วอยู่ในโฟลเดอร์โปรเจกต์ พิมพ์:

```bash
docker compose down

python rag/pipeline.py --reset
# คำสั่งนี้จะ: ลบ chunks.jsonl และ qdrant_db เดิมทิ้ง
# แล้วตัด chunk ใหม่ทั้งหมด (AAFP/URI/Dose) → แปลงเป็นเวกเตอร์ใหม่ทั้งหมด

docker compose up --build
```

หรือจะทำทีละขั้นตอนก็ได้:

```bash
python rag/pipeline.py --chunk-only   # สร้างแค่ chunks.jsonl (เปิดดูตรวจสอบได้ก่อน)
python rag/pipeline.py --embed-only   # เอา jsonl ที่มีอยู่ไปแปลงเป็นเวกเตอร์อย่างเดียว
```

ถ้ามี PDF เล่มใหม่ทั้งเล่ม → ต้อง OCR ด้วย `pdf_to_md.py` ให้ได้ไฟล์ `.md` ก่อน แล้วค่อยรัน `--reset`

อ่านรายละเอียดเรื่อง Strategy C เพิ่มเติมได้ที่: `readme/STRATEGY_C_explained.md`

```mermaid
  F --> R["5. จัดอันดับใหม่ (Rerank)\nปกติใช้ AI (gemini-3.1-flash-lite) จัดอันดับ\nถ้า AI ใช้ไม่ได้ จะสลับไปใช้วิธี BM25 แทน"]
  R --> T["6. เลือกเอาแค่ 5 ชิ้นที่ดีที่สุด ส่งให้ AI เป็นข้อมูลอ้างอิง"]
  T --> A["7. AI (Gemini) ตอบคำถาม\nตามกฎที่กำหนดไว้ + แนบแหล่งอ้างอิง"]
```

**พูดง่ายๆ:** ถาม → ไปหาชิ้นข้อมูลที่เกี่ยวข้องจาก 3 เอกสาร (ดึงมาก่อนเอกสารละ 8 ชิ้น) → จัดอันดับใหม่ว่าชิ้นไหนตรงคำถามที่สุด → เหลือแค่ 5 ชิ้นที่ดีที่สุด → ส่งให้ AI อ่านแล้วตอบ

#### โมเดล AI ที่ใช้ + ค่าตั้งต่างๆ (knobs)

| ใช้ทำอะไร | ใช้โมเดล/ค่าอะไร | ไปแก้ที่ไหน |
|-------|-----|--------|
| แปลงข้อความเป็นเวกเตอร์ | `models/gemini-embedding-001` | `config.py` → ตัวแปร `EMBED_MODEL` |
| ตอบคำถามในแชท | `models/gemini-3.1-flash-lite` | ตัวแปร `CHAT_MODEL` |
| จัดอันดับใหม่ (rerank) แบบปกติ | ใช้โมเดลตัวเดียวกับตอนตอบแชท | ตัวแปร `RERANK_MODE=llm` |
| จำนวนชิ้นที่ดึงมาต่อเอกสาร 1 แหล่ง | **8 ชิ้น** | ตัวแปร `PER_SOURCE_TOP_K` |
| จำนวนชิ้นสุดท้ายที่ส่งให้ AI ตอบ | **5 ชิ้น** | ตัวแปร `TOP_K` |
| วิธีจัดอันดับ (เลือกได้) | `llm` (ใช้ AI) / `bm25` (นับคำ) / `vector` (ระยะเวกเตอร์) | ตั้งค่าผ่าน env `RERANK_MODE` |

โค้ดหลักของส่วนนี้อยู่ที่ `backend/rag_engine.py` (prompt ก็อยู่ในไฟล์นี้เช่นกัน)

### สิ่งที่ควรโฟกัสถ้าจะแก้ prompt (แก้ปัญหาข้อ 3, 4, 7 ด้านบน)

1. **บังคับให้มี URL ในการอ้างอิง** เวลาที่ AI ใช้ความรู้จากภายนอก (แก้ปัญหาข้อ 3)
2. **แยกให้ชัดเจน** ระหว่าง "ข้อมูลจาก Guideline ในระบบ" กับ "ความรู้ทั่วไปของ AI" — ห้ามปนกันโดยไม่บอกผู้ใช้ (แก้ปัญหาข้อ 4)
3. **เมื่อมีข้อมูลทั้ง URI และ AAFP อยู่ในบริบทเดียวกัน** ให้ AI เทียบความต่าง เช่น ระยะเวลาการรักษา (แก้ปัญหาข้อ 7)
4. **เรื่องขนาดยา (Dose):** ย้ำในกฎว่าตัวเลขขนาดยาต้องมาจากข้อมูลที่ค้นเจอเท่านั้น **ห้าม AI แต่งตัวเลขขึ้นเอง**

หลังแก้ `SYSTEM_PROMPT` เสร็จ: เพราะโฟลเดอร์ `backend/` ผูก (mount) กับ container ไว้แล้ว → แค่ `docker compose restart` หรือรอระบบโหลดใหม่ แล้วลองถามคำถามใหม่ได้เลย
**ไม่ต้อง** รัน `pipeline.py --reset` (เพราะการแก้ prompt ไม่เกี่ยวกับการแปลงข้อมูลเป็นเวกเตอร์)

### สิ่งที่ต้องรู้เพิ่มเติม
- ความถูกต้อง Accuracy & Guideline -> ต้องมาก่อน 
    - วินิจฉัย เลือกยา First-line ยาทางเลือกกรณีแพ้ยา ขนาดยา และระยะเวลารักษา ถูกต้องตาม Guideline สม่ำเสมอทุกเคส
    - ข้อมูลประเภทยาตามกฎหมายถูกต้อง (ยาอันตราย vs ยาควบคุมพิเศษ) เพราะกระทบการตัดสินใจจ่ายยาโดยตรง
    - คำแนะนำไม่ขัด Guideline โดยเฉพาะประเด็นความปลอดภัยในเด็กเล็ก
    - ดึงข้อมูลถูกโรค ถูกช่วงวัยผู้ป่วย และถูกหัวข้อใน Guideline
    - ระบุชื่อตัวยาชัดเจนตั้งแต่คำตอบแรก ไม่ตอบแค่กลุ่มยา
    - เข้าใจคำที่สะกดต่างหรือใช้คำต่างกันเล็กน้อยแต่ความหมายเดียวกัน


- การคำนวณขนาดยา (Dose Calculation)
    - แสดง Dose เป็นช่วง Min–Max เต็มช่วงตาม Guideline
    - คำนวณขนาดยาตามน้ำหนักตัวให้อัตโนมัติ และคำนวณออกมาเป็นช่วงขนาดยา
    - ระยะเวลารักษาจำเพาะกับผู้ป่วยแต่ละราย (เช่น ตามอายุ)
    - ยืดหยุ่นตามความรุนแรงของอาการ และแนะนำในรูปแบบใช้จริง เช่น "Amoxicillin 500 mg ครั้งละ 1 เม็ด วันละ 2–3 ครั้ง"
    - ต่อยอด Drug Calculator: เลือกความแรงยาน้ำที่มีในร้าน และคำนวณปริมาณที่ต้องกินเป็น mL
    

    - และเพิ่มเติมคือการ scale ในเคสที่มีการถามเพิ่มเติม เช่น ตอนแรกถามขนาดยาของเด็ก สามารถตอบ ____ จนาดได้ถูกต้อง และหากถามต่อหากเป็นผู้ใหญ่ ขนาดต้องกลายเป็น ____ (การคำนวณต้องเพิ่มขึ้นตามไม่ใช่ยึดตามข้อมูลเก่าเพียงอย่างเดียว) -> การคำนวณตรงนี้เป็นเหมือนเชิงข้อสอบ ต้องอิงหลักๆคืออายุและ Guideline หลัก (ห้ามมั่ว และพยายามอิงตามที่มี) แค่หากเจอเคสที่อายุต่ำลงหรือถามเชิงอายุมากขึ้น ขนาดยาต้องปรับตาม และปรับอย่างเหมาะสมตามเกณฑ์ทางเภสัชกรรมด้วย ไม่ใช่ปรับเรื่อยเปื่อยหรืออยู่ที่เดิม


- การอ้างอิงแหล่งข้อมูล (References)
    - ทุกคำตอบต้องมีเรฟ Guideline กำกับ พร้อมชื่อเอกสารและเลขหน้าที่ถูกต้องแม่นยำ
    - แยกชัดว่าข้อมูลมาจาก Guideline หรือแหล่งภายนอก ถ้ามาจากอินเทอร์เน็ตให้แปะ URL/ลิงก์ที่มา
    - ค้นเจอข้อมูลที่มีอยู่ใน Guideline ได้ครบ ต้องไม่ตอบว่า "ไม่มี" ทั้งที่มีอยู่จริง
    - ใช้แหล่งอ้างอิงครบทุก Guideline (ทั้งแนวทางของไทย และ AAFP)
    - [แผนในอนาคต] แสดงคำแนะนำเปรียบเทียบจากหลาย Guideline ให้เภสัชกรใช้ประกอบการตัดสินใจ -> นี่คือจุดที่ต้อง optimize เพิ่มต่อจาก demo ที่ไม่มี (คุณออกแบบได้เลยแต่ต้องถูกต้องด้วย)


    - เพิ่มเติมคือ ส่วนการอิงใน Guideline เน้นมากให้ปฏิบัติตามด้านบน ส่วนนอก Guideline **กรณีที่ถามนอกเรื่อง Guideline หากมีการอ้างอิงให้อ้างโดยแนบ URL จาก External มาด้วย โดย URL ตัวนั้นไม่ใช่แค่ URL ที่กดเข้าไปแล้วเป็นหน้า Landing Page เปล่าๆหรือหน้าต้อนรับ มันต้องเป็นหน้าที่มีเอกสารระบุอยู่ชัดเจน และหากเป็นไปได้ระบุตำแหน่งของตรงนั้นด้วย อาจเป็นเลขหน้าหรือข้อความเป็นต้น (หากฉันกด link เข้าไปแล้ว ฉันต้องเจอข้อมูลที่ตรงกับสิ่งที่ LLM ตอบ ทันที (Highlight ด้วยหากทำได้))**
    - เน้นทั้งการอ้างอิงภายใน Guideline และนอก Guideline


- การซักประวัติและเหตุผล (History Taking & Clinical Reasoning)
    - ซักประวัติครบถ้วน พร้อมเหตุผลกำกับทุกคำถามว่าถามไปเพื่ออะไร (เช่น "เป็นกี่วัน — เพื่อประเมินความรุนแรง") และถามประวัติแพ้ยาพร้อมลักษณะอาการแพ้ -> ตรงนี้อาจต้องจัด Format ให้ดีเพื่อความอ่านง่าย (UX/UI)
    - อธิบายเหตุผลประกอบทุกการตัดสินใจ ทั้งการเลือกยา การไม่จ่ายยาปฏิชีวนะ และการส่งต่อโรงพยาบาล
    - เหตุผลทางคลินิกถูกต้องตามหลักวิชาการ ไม่ใช่แค่คำตอบสุดท้ายถูกอย่างเดียว
    - เข้าใจบริบทว่าผู้ใช้คือ "เภสัชกร" เน้นเสนอแนวทางรักษาสำหรับร้านยา และแนะนำ "ไปพบแพทย์" เฉพาะกรณีมี Red Flag เท่านั้น
    

    - เพิ่มเติมคือการซักประวัติ ต้องเกิดขึ้นในเคสที่ข้อมูลหรือบริบทไม่ครบ เช่นสำคัญๆคือ อาการที่พบ อาการแพ้ยา ช่วงเวลาที่เป็น มีไข้หรือไม่เป็นต้น (อาจมีมากกว่านี้ได้แต่ให้ถูกต้อง)


- รูปแบบคำตอบ (Answer Format)
    - เรียงคำตอบเป็นมาตรฐาน 5 ขั้น: สรุปอาการ → วินิจฉัยเบื้องต้น (พร้อม range ความน่าจะเป็นของโรค) → การรักษา (แยกยาปฏิชีวนะ / ยาตามอาการ) → คำแนะนำดูแลตัวเอง → Red Flags
    - คงการแบ่งบูลเล็ทการตัดสินใจของเภสัชกรพร้อมเหตุผลไว้อย่างชัดเจน


    - โดย Format ตามด้านบน โดย Format นี้หากเป็นไปได้พยายามจัดการด้าน Highlight คำ, Layout, Component ยังไงก็ได้ให้อ่านง่ายสบายตา Focus ได้ง่าย (เน้นมากสำหรับ UI) -> ฉันให้สิทธิ์คุณออกแบบ UX/UI ตรงนี้ได้เต็มที่ โดยที่ยัง Format เป็นไปตามด้านบน


- การจัดการบทสนทนา (Conversation Context)
    - จำบริบทบทสนทนาต่อเนื่อง แยกเคสเดิม-เคสใหม่ได้ถูกต้อง รวมถึงโรคใหม่ในผู้ป่วยคนเดิม (กรณีแชทเดิม)
    - ตอบคำถาม Follow-up เฉพาะประเด็นที่ถามเพิ่ม ไม่ตอบข้อมูลเดิมซ้ำทั้งหมด -> หากถามเพิ่มเติมต่อจาก Format ล่าสุดที่ LLM ให้ ต้องตอบเพิ่มเติมได้ ไม่ใช่เอา Format เดิมกลับมาตอบอีกครั้ง (ไม่เป็นธรรมชาติและหลุดประเด็น)
    - จำแนกประเภทคำถามแม่นยำ (เคสผู้ป่วย / ความรู้ทั่วไป / นอกขอบเขต)
    - ไม่ Hallucinate ไม่สรุปหรือเติมข้อมูลที่ผู้ใช้ไม่ได้ให้มา -> สำคัญมาก เช่น ยังไม่ได้บอกอาการแพ้ แต่ LLM สรุปว่ามีอาการแพ้ยา ห้ามและอื่นๆที่ใกล้เคียง
    - คงความระมัดระวังต่อเคสหลอก (Trick questions) และการแยกอาการแพ้ยาจริง/ไม่จริงไว้
    

    - เพิ่มเติมคือการจัดการเคส Negative หากเจอเคสที่ Negative เช่นเป็นโรค ___ แต่ขอยาปฏิชีวนะ หรืออะไรที่เป็นเชิง Negative, LLM ต้องปฏิเสธและแนะนำทางเลือกที่ดีกว่าและ**ถูกต้อง**ตาม Guideline หรืออ้างอิงได้



จากด้านบนเป็นสิ่งที่ต้องจูนและ Checklist ออกมา


Permission: ฉัน Allow หมดทุกคำสั่งสำหรับการ Optimization ในครั้งนี้ หนึ่งสิ่งที่ฉันห้ามคือ การแก้ไข vector database หรือปรับหน้าตา chunking (ยุ่งกับต้นทาง) แค่นั้นพอ


ควรสามารถ Optimize ได้เลยสำหรับ Kernel ไม่ว่าจะดึงข้อมูลออกมาดูก่อนและเช็คความถูกต้อง เพื่อปรับจูน


Intial Flow:
1. อ่านและเข้าใจ prompt ด้านบน
2. เข้าไปดูตัวอย่างข้อมูล จากการลอง Retreieve ออกมาดู (ลองดูเท่านั้น) จากทั้ง 3 ข้อมูลหลัก
3. ไปที่ data_for_prompt_optimization/ แล้วอ่าน Feedback จากทุกไฟล์
4. เช็ค Checklist ให้ชัวร์ว่าต้องทำอะไรบ้าง
5. วางแผนการ Optimization + Refactor LLM ในครั้งนี้ หา Best Solution หรือทดลองจนกว่าจะเจอจุดต่ำสุดได้
6. แก้ไขโค้ดตาม Best Solution ที่ออกแบบในแผน โดยการเขียนโค้ด ควรเขียนให้ clean, flexible เน้นการเป็น dynamics programming, ยังทำ Vanila RAG อยู่ (ไม่ติดปัญหา dependencies เหมือน langchain), งดอิโมจิ, เน้น Production Grade MVP
7. หลังได้โค้ด Best Solution ตรวจสอบความถูกต้องด้านข้อมูลที่ได้รับจาก LLM **เน้นมาก**
8. ลองทำตาม checklist แล้วไล่เช็คทีละข้อ โดยต้อง pass 100% เท่านั้น และหาก optimize เพิ่มเติมในจุดที่ฉแันไม่เห็นได้ยิ่งดี
9. เขียนคำตอบ สรุปผลการทำ และสรุปเช็ค list ผลคะแนนต่างๆลง Jack_prompt/result/ ในไฟล์ชื่อ result_opt_1.md (สร้างกณณียังไม่มี) -> ไฟล์ตัวนนี้ต้องบอกรายละเอียดที่คุณทำไปทั้งหมดรวมถึง เช็ค list ว่าผ่านไม่ผ่าน และ อีกตัวอย่างการลองทำ (สัก 12 เคส ทั้ง Positive, Negative, ข้อมูลไม่ครบ, เคสที่ต้องอ้างอิงจากด้านนอก) (เคสละ 3)
10. เช็ค SyntaxError, NameError, IndexError, KeyError หรือ Error ที่เกี่ยวข้องต่างๆภายในระบบ
11. ส่งมอบให้ฉัน
