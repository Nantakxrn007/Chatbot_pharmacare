    - แต่หลักๆฉันอยากให้คุณ Apply & Implement ดูก่อนว่าเข้ากับระบบปัจจุบันที่เป็นอยู่หรือไม่อย่างไร หากนำเข้ามาใช้แล้วดีกว่าเดิม (ใช้) แต่หากนำมาใช้แล้วแย่กว่าเดิม (ไม่ควรใช้) อาจประยุกต์บางจุดพอ

- เคสอย่างเช่น "น้ำหนัก 15 kg ปวดหูข้างเดียว มีไข้ 38 องศา ไม่มีหนองไหลจากหู เคยได้ amoxicillin เดือนก่อน ไม่มีแพ้ยา ไม่สะดวกสังเกตอาการ" -> จริงๆมันดีมากละนะแต่ส่วนของ "การรักษาด้วยยา" คือตอนนี้มันอ้างอิงเฉพาะ URI ในขณะที่ AAFP ก็มีโดยเฉพาะเวลาจำเพาะ (AAFP อยู่ หน้า 6(หน้า pdf) Table 4) โดยนอกจากข้อมูลในตารางแล้ว มันยังมี desc ด้านล่าง "*—Adults with mild to moderate infection can be treated for five to seven days, patients with severe illness should be treated for 10 days.
†—Children < two years or ≥ two years with severe symptoms should receive a 10-day course of oral antibiotics. Children two to five years of age
with mild to moderate symptoms should be treated for seven days, children ≥ six years with mild to moderate symptoms should be treated for five
to seven days."
    - จริงๆมันอ้างถูกต้องแล้วแหละ แต่มันขาด "อ้างอิง AAFP" เคสทำนองนี้ที่มันมีคล้ายกัน 2 จุด แล้วคุณต้องอ้างอิงทั้ง 2 เอกสาร -> ให้หยิบตัวอ้างอิงมาทั้งคู่เลย (จะได้รู้ว่าเห็ฯทั้งหมดแล้ว ไม่ใช่เห็นอันใดอันนึงแล้วตอบ เพื่อให้ user มั่นใจ)
    - ส่วนเคสที่เรื่องเดียวกัน (ส่วนใหญ่เป็นเด็ก) แล้วข้อมูล conflict -> ให้ LLM นำเสนอทั้ง 2 เลย เช่น จาก Guide abc แนะนำเป็น dfsds ในขณะที่ Guideline B แนะนำ  กดเกดเ ซึ่งแตกต่างกันในเชิง เดกเ เภสัชกรอาจจำเป็นต้อง บลาๆ และผมแนะนำว่า หเกเ บลาๆ อะไรทำนองนี้
    - ฉันมองว่าเคสแบบนี้คือความถูกต้องของอ้างอิงเลยและความครบถ้วนด้วย -> มันใกล้ถึงฝันละ (90%) -> เหลือ detail เล็กๆน้อยๆที่พูดไป แก้ไข ทำให้ดีขึ้น และเคสที่ใก้ลเคียงทำนองนี้ด้วย

- Cold Start ตอนเริ่มต้นใหม่ คือบางทีฉันแค่พิมพ์มั่วๆหรือพิมพ์ทักเช่น ดีจ้า, ทดสอบๆ, sdfdsf, g543f ตอนเริ่มแชทใหม่ -> ไม่ควรช้า นอกจากหากเปิดแชทใหม่แล้วใส่เคสเลยหรือเป็นคำถามจริงจัง แบบนี้ยังพอเข้าใจได้
- อีกปัญหาด้าน Context Management ที่ฉันเจอ ฉันลองถาม "อายุ 50 ปี ปวดหน้าผาก ปวดโหนกแก้มซ้าย น้ำมูกเหลืองข้น ไข้ 37.9 องศา มาได้ 11 วัน ไม่ได้ดีขึ้นเลยตั้งแต่ต้น ไม่มีประวัติแพ้ยา อยากได้ยา Penicillin V" -> ตอบได้ดีเลย อ้างอิงถูกต้อง (ไม่มีการอ้างอิง URI เพราะอายุ 50 ปี และตำแหน่งที่อ้างอิงถูกต้อง) แต่ปัญหาที่เจอคือ หลังจากนั้นฉันถาม "ปัจจัยเสี่ยงที่ทำให้เกิดโรคมีอะไรบ้าง" ยังเป็นปัญหาอยู่ จากคำตอบนี้
    - ในเคสนี้ถือเป็น ประเภท 1: ความรู้ทั่วไปในคู่มือ ครับ ปัจจัยเสี่ยงที่ทำให้เกิดโรคติดเชื้อทางเดินหายใจในเด็กตามแนวทางเวชปฏิบัติ มีรายละเอียดดังนี้ครับ:

1. ปัจจัยเสี่ยงต่อโรคหูชั้นกลางอักเสบเฉียบพลัน (AOM)

ปัจจัยด้านประวัติและสุขภาพ: มีประวัติเจ็บป่วยทางระบบหายใจบ่อย, มีประวัติครอบครัวป่วยเป็นหูชั้นกลางอักเสบ, มีภาวะกรดไหลย้อน, ปากแหว่งเพดานโหว่, กลุ่มอาการดาวน์ (Down syndrome) และมีความผิดปกติของภูมิคุ้มกัน 📄 URI เด็ก 2562 p.50
ปัจจัยด้านสิ่งแวดล้อมและการเลี้ยงดู: การเลี้ยงดูในสถานเลี้ยงเด็กอ่อน, การใช้จุกนมปลอม, การอยู่ในสิ่งแวดล้อมที่มีควันบุหรี่ และการมีอายุน้อย 📄 URI เด็ก 2562 p.50
2. ปัจจัยเสี่ยงต่อโรคหวัด (Common cold)

ปัจจัยด้านฤดูกาล: มักพบในฤดูหนาวหรือช่วงที่มีอากาศเย็นและความชื้นสัมพัทธ์ต่ำ เนื่องจากอุณหภูมิเอื้อต่อการเติบโตของไวรัส และภาวะเยื่อบุโพรงจมูกแห้งทำให้ติดเชื้อได้ง่ายขึ้น 📄 URI เด็ก 2562 p.16
ปัจจัยด้านสถานที่: เด็กที่อยู่ในสถานรับเลี้ยงเด็กมีโอกาสติดเชื้อหวัดได้มากขึ้น 📄 URI เด็ก 2562 p.16
3. ปัจจัยเสี่ยงต่อภาวะฝีหลังคอหอย (Retropharyngeal abscess)

การติดเชื้อลุกลาม: เกิดจากการติดเชื้อบริเวณใกล้เคียง เช่น คออักเสบ, ทอนซิลอักเสบ, ไซนัสอักเสบ หรือการติดเชื้อของเหงือกและฟัน 📄 URI เด็ก 2562 p.60
การบาดเจ็บหรือสิ่งแปลกปลอม: การได้รับอุบัติเหตุจากสิ่งของทิ่มตำในช่องปาก (เช่น อมยิ้ม, ของเล่น, ก้างปลา) หรือการใส่อุปกรณ์ทางการแพทย์ผ่านทางหลอดคอ เช่น ท่อช่วยหายใจ, การส่องกล้อง หรืออุปกรณ์ทำฟัน 📄 URI เด็ก 2562 p.60
    - ปัญหาก่อนหน้าคือ เขาพูดถึง (Acute Bacterial Rhinosinusitis: ABRS)  แต่อันนี้มาพูดถึง AOM และยังเป็นเคสเด็ก (ดูจากอ้างอิง) -> คนละเรื่องเลย
    - ต้องปรับตรงนี้และไม่ใช่เคสนี้ มันควรทุกเคสที่เกิดเหตุการณ์แบบนี้ แล้วพยายามอย่า fig ตรงๆลงไปใน prompt เพราะไม่งั้นมันก็จำแค่เคสใดเคสนึง (ประยุกต์ Context Management ด้านบนแล้วลองดูว่าแก้ได้หรือไม่)
    - ปัญหายิ่งคุยเยอะในแชทเดิมแล้วเริ่ม Hallucination หรือ Conversation Driff ยังมีอยู่และเป็นปัญหาที่กวนใจมาก -> พยายามแก้จุดนี้ให้ได้
- "อายุของผู้ป่วย: เพื่อเลือก Guideline ที่ถูกต้อง (เด็กใช้ URI เด็ก 2562, ผู้ใหญ่ใช้ AAFP)" ฉันเจอเคสสทำนองนี้ จริงๆ AAFP ก็มีเด็กด้วย คุณฟันธงแบบนี้อาจไม่ดีเท่าไร อาจพิจารณาการอ้างอิงตรงนี้และการใช้คำ
    - ฉันแนะนำให้คุณลองดึง chunk ออกมาดูทั้ง 2 แหล่งโดยเฉพาะในเรื่องเดียวกัน (เด็ก vs เด็ก)
- ฉันเจอเคสการคำนวณ mL
    - "ข้อแนะนำเพิ่มเติมสำหรับเภสัชกร:

    - เป็นไปได้ โชว์การคำนวณ ตรง mL ออกมาเลย step-by-step -> เพื่อความชัดเจน (ทั้ง min-max เอามาหมด) (logic ตอน convert มาเป็น mL)

- ตอนจากเคสการคำนวณต่างๆ ฉันอยากให้เพิ่มหลังจากการคำนวณ คือ Dose, mg, mL เช่น 10 mg/kg, 200 mg, 8.3 mL (ตัวอย่าง) โดยขึ้นอยู่แบบแต่ละเคส คือไม่ต้องตีเส้นตารางก็ได้ (เอาเท่าที่ทำได้) อาจจะแบ่งเป็นรูปแบบ วางเว้น space ให้เท่ากัน (ให้ดูเหมอนตารางแต่ไม่มีเส้น) -> กรณีเราสร้าง ตารางเองไม่ได้ (อาจลองสร้างแบบ markdowm ก่อน)
    - อย่าลืมเรื่องระยะเวลาการให้ยาด้วย (อ้างอิงต้องดูเช่นกัน รวมถึงเวลาจำเพาะ (หากเคสเด็กแล้วเรื่องเดียวกัน หยิบมาทั้ง 2 เช่น URI อาจบอก 7-10 days แต่ AAFP บอก 7 Days -> หยิบมาคู่แล้วอ้างอิงว่ามาจากไหน))
- หัวข้อ การคำนวณขนาดยา (Dose Calculation) -> ระยะเวลารักษาจำเพาะกับผู้ป่วยแต่ละราย (เช่น ตามอายุ) -> Calculate หรือ Scale ในเคสที่ user ถามต่อเช่น (หากเขาอายุมากขึ้นล่ะ, แล้วถ้าอายุ 50 ปี (มากขึ้นกว่าเดิม), หรือน้อยลงเป็นต้น) -> ให้มันคำนวณตรงนี้ (และ validate ด้วย ว่าสิ่งที่คำนวณออกมา ถูกต้องจริง) หลักๆคือคำนวณได้ scale ต่อได้ + ถูกต้อง + ครบถ้วน + อ้างอิง
- ในหัวข้อ การอ้างอิงแหล่งข้อมูล (Reference) ในเคสที่อ้างอิง Guideline ที่มีอยู่ ฉันต้องการให้แสดงคำแนะนำเปนียบเทียบจากหาย Guideline ให้เภสัชกรใช้ประกอบการตัดสินใจ (ทำเป็นตารางก็ดี) -> ซึ่งมันจะะเชื่อมกับสิ่งที่ฉันบอกก่อนหน้าคือ เคสที่มีการอ้างอิงเหมือนกันหรือเคสที่เรื่องเดียวกันแต่การแนะนำต่างกันเล็กน้อย (ใกล้เคียงกันแต่ไม่เหมือนกัน) -> ทำตรงนี้ได้จะดีมาก
- Latency อยากได้ไวขึ้นอีกหน่อย (ประสิทธิภาพต้องไม่ลดลงด้วย)



หลักๆที่ต้องแก้ไขใน Step 8 นี้

หลักๆคือคุณต้องเข้าใจ Context เดิมก่อน, ข้อมูล, สิ่งที่ Optimize ไปก่อนหน้า, ปัญหาที่ยีงเป็นอยู่


โดยเงื่อนไขเดิมที่ฉันยังกระชับอยู่และคุณต้องปฏิบัติตามอย่างเด็ดขาด
- **ห้าม** แก้ไข chunk หรือลบหรืออะไรก็แล้วแต่ในหมวดแก้ไข ฉันอนุญาติแค่ดึงมาตรวจสอบหรือดึงมาดูเพื่อใช้อ้างอิงหรือเพื่อเข้าใจข้อมูลมากขึ้นเท่านั้น
- **เน้นย้ำ** ข้อมูล**ต้องถูกต้อง** + **ครบถ้วน** (context detail เล็กน้อยต้องหยิบมาให้หมด ไม่ใช่หยิบแค่ตัวหลัก เอา detail มันมาด้วย) + **อ้างอิงถูกต้อง ครบถ้วน** และเคสที่มีเปรียบเทียบ หยิบมาทั้งอ้างอิงแล้วแสดงให้เห็นแนวทางแต่ละไกด์ไลน์ได้อย่างชัดเจน (กรณีในเคส Guideline หากนอก Guideline ตัวอ้างอิงห้ามนำมาตอบปนกันหรือหรืออ้างอิงรวมกัน) 
- Optimization ครั้งนี้ควรเป็ฯครั้งสุดท้าย


โดยปรับปรุงต่อจากด้านบนเป็นไปได้คือ LLM ควร dynamics มากที่สุด (พยายามอย่า fig ข้อมูลลง prompt ไม่งั้นมันก็ตอบได้แค่เคสนั้นๆ มันควรหยืดหยุ่นได้มากกว่านั้น) + ครอบคลุม Edge Case

 
การเขียนโค้ดเน้นเชิงทางการ, clean code, less comment, no emoji เหมือนเดิม


Flow:
1. เข้าใจ Context ทั้งหมดก่อน รวมถึงสามารถทดสอบเพื่อเพิ่มความเข้าใจได้
2. ออกแบบ Solution -> ทดลองจนได้ Best Solution ที่ Optimal
3. Validate Best Solution ตัวนั้น ทั้งความถูกต้องต่างๆ ตาม feedback ต่างๆ รวมถึง checklist
4. ทดสอบจริง (ในเคสที่เคยเป็นปัญหามาทั้งหมด)
5. Validate เคสทดสอบจริง (ของเก่าจากข้อ 4) -> ควร Pass 100%
6. Test with New Case -> Generate ขึ้นมาสัก 12 Cases (ค้ลายกับของเก่า แต่ปรับให้มันยืดหยุ่นมากขึ้น)
7. Validate -> ควร Pass 100%
8. ตรวจสอบเคสที่ ถามคำถาม Following แล้วมัน Hallucination หรือไม่ หรือเกิด Conversation Drift -> ไม่ควรเกิดขึ้นเลย (เนื่องจากด้านนี้อาศัยความถูกต้อง มันจึงห้ามเกิดขึ้นเลย) -> หากมี resouce จำกัดพยายามออกแบบให้มัน scale ให้ได้
9. Validate ตรวจสอบข้อ 8 อีกที
10. หากผ่านทั้งหมด Validate ตรวจสอบด้วยรวมด้านความถูกต้องของคำตอบบ LLM และปัญหาของระบบที่ไม่ควรเกิดขึ้น เช่น Syntax Error, NameError, IndexError, KeyError, AttributeError ต่างๆและอื่นๆ
11. ตรวจสอบตาม Checklist (Pharmacy Bot Evaluation Checklist.md) ว่าผ่านทั้งหมด และมีประสิทธิภาพด้วย (ทดสอบกับเคสจริง ไม่ใช่เคสขึ้นมาเอง) -> Checklist ต้อง Pass 100% 
12. เขียน result_optt_8.md ใน Jack_prompt/result ว่า การแก้ไขครั้งนี้เน้นเรื่องอะไร ปัญหาที่ยังมี (มีตัวอย่างด้วยก็ดี) solution ที่ออกแบบ และประสิทธิภาพการทำงาน ดีขึ้นอย่างไร, checklist ตรวจสอบว่าผ่านหมด พร้อม ตัวอย่างอ้างอิง ว่าทำได้จริง, validate ความถูกต้องทั้งหมด และสรุปผล + แนวทางต่อยอดในอนาคต
